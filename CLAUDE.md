# CLAUDE.md — operating instructions for this repo

This file is read automatically by Claude when it is pointed at this repository.
It tells Claude how the Project H.O.O.D. website is built and how to make changes
safely. Maintainers: you generally don't need to read this — it's for Claude.
Humans should start with **UPDATING_WITH_CLAUDE.md**.

---

## What this is

The static HTML source for **projecthood.org** (Project H.O.O.D., a Chicago
nonprofit). Plain HTML + one CSS file + a little JS. No framework, no build step
to *view* — every `.html` file renders on its own.

- **GitHub repo:** `projecthood-site/projecthood`
- **Working branch:** `staging` — make all edits here.
- **Deploy:** GitHub Pages serves from the **`main`** branch (root). Changes go
  live only after `staging` is merged into `main`. Production `projecthood.org`
  is still on Squarespace pending DNS cutover (see `LAUNCH_RUNBOOK.md`).

## Golden rules

1. **Edit on `staging`.** Never commit directly to `main`. Merging to `main` is a
   deliberate "publish" step the human approves.
2. **Confirm before writing to GitHub.** Show the human the diff and get a clear
   yes before committing or pushing. Pushing and merging are publish actions.
3. **Use brand tokens, never raw hex.** All colors come from CSS variables in
   `css/styles.css` (see palette below). Don't introduce new hardcoded colors.
4. **Don't hand-edit the nav or footer in individual pages.** They are templated.
   See "Shared nav/footer" below.

## File layout

```
*.html              One file per page (edit page content directly here)
css/styles.css      Design system — brand colors, typography, components
js/main.js          Mobile nav, external-link safety, active-nav highlighting
img/                Logos + acronym marks
_build.py           Regenerates shared <head>/nav/footer across all pages
docs/               PDFs served by the site
```
Page-by-page purpose and content owners are in `SITEMAP.md`.

## Shared nav/footer (important)

The header/nav and footer are identical across pages and generated by
`_build.py` from its HEAD / FOOTER templates. To change anything in the nav or
footer (links, logo, newsletter block, copyright):

1. Edit the template strings in `_build.py`.
2. Run `python3 _build.py` to regenerate every page.
3. Review the diff, then commit.

For **page body content** (headlines, paragraphs, sections, images, videos on a
single page), edit that page's `.html` directly — no build step needed.

## Brand kit (PH Visual Identity Guide V1)

The palette in `css/styles.css` matches the official guide exactly. Use the
tokens — do not hardcode these values:

| Token | Hex | | Token | Hex |
|---|---|---|---|---|
| `--green` | #006a63 | | `--purple` | #712f6b |
| `--red` | #ef4032 | | `--offwhite` | #e2e4e1 |
| `--blue` | #3face1 | | `--black` | #231f21 |
| `--lightblue` | #91c7ec | | `--white` | #ffffff |
| `--yellow` | #fbe14c | | `--gold` | #8a6d00 (darkened yellow for accessible text/accents on light) |

Fonts are Oswald (display / headings), Zilla Slab (body), Montserrat — these are
licensed web substitutes for the print-only Akkordeon Nine and Freight Micro in
the guide. The full guide PDF lives in the parent project folder.

## Conventions

- **Videos:** embed YouTube with the responsive `.video-frame` wrapper:
  `<div class="video-frame"><iframe src="https://www.youtube.com/embed/ID" title="..." allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture; web-share" referrerpolicy="strict-origin-when-cross-origin" allowfullscreen></iframe></div>`
- **Section backgrounds** alternate (`bg-offwhite`, `bg-red`, `bg-green`, plain
  white). Keep adjacent sections from repeating the same background.
- **Forms / external destinations** (do not swap these for on-site alternatives
  without asking):
  - Donate → NetworkForGood (`projecthood.networkforgood.com`)
  - Walk With Us / WAA → Tiltify
  - Volunteer + Contact → Google Form embeds
  - Construction cohort application → Google Form (workforce-development.html;
    currently a `[GOOGLE FORM APPLICATION URL]` placeholder awaiting the real link)
- **Preview state:** pages still carry a "PREVIEW BUILD" banner and
  `noindex,nofollow`. These are removed at launch via the `_build.py` HEAD
  template — see `LAUNCH_RUNBOOK.md`. Don't remove them ad hoc.

## After editing — verify

- Open the changed page(s) in a browser and check layout/links.
- Confirm no raw hex colors were introduced (grep for `#` in inline styles).
- Confirm HTML tags are balanced.
- If `_build.py` was touched, re-run it and confirm all pages regenerated.

## Publish flow

1. Commit to `staging` with a clear message (after human confirms the diff).
2. Push `staging` to GitHub.
3. To go live: open a PR / merge `staging` → `main` (human approves). GitHub
   Pages rebuilds from `main`.
